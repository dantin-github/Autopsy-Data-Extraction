# 4.3.5 Upload and Proposal Configuration 的设计逻辑与合理性分析

## 一、功能定位：解决什么核心问题

4.3.3 节描述的结构化报告生成模块能够在 Autopsy 内产生一份包含案件元数据、操作日志、数据源信息、文件清单和聚合哈希的完整取证记录（`case_data_extract.json`）。然而，这份报告生成之后仅停留在取证人员的工作站本地。要使该报告进入区块链辅助监管链工作流——即将其完整性承诺锚定到区块链上——必须建立一条从 Autopsy 到 gateway 再到区块链的数据通道。

传统做法存在明显缺陷：取证人员需要手动将报告文件从 Autopsy 目录复制出来，通过独立客户端上传到后端系统，并手动记录上传结果。这一过程割裂了取证分析与完整性锚定两个环节——报告生成时间、上传时间、链上确认时间之间缺乏自动关联，审计记录依赖于人工操作纪律，且没有任何机制阻止"生成本地报告后忘记上传"的情况。

本功能的设计目标是：在 Autopsy 的报告向导内部嵌入一个配置层，使取证人员在完成本地报告生成后，能够通过同一个界面选择将报告提交到 gateway（初始提交）或发起对已登记记录的修改提案（修改提案），并将上传结果自动记录为本地回执。核心意图是让"上传"成为报告生成工作流的自然延伸，而非一个外部的、与取证工具分离的操作。

与 4.3.3 节报告生成模块的关系需要在此明确：4.3.3 的产物是本地的结构化报告文件，是取证工作站上的主要证据记录；本节描述的产物是一个 HTTP 请求体，它将那份本地报告作为 `caseJson` 字段包裹在内，并附加提交所需的元数据字段。两者不是两种独立的报告格式，而是**源与提交副本的关系**——本地报告生成必须先于上传执行，上传内容是本地报告的打包 submission 形式。

## 二、模块运行机制

本节描述取证人员在 Autopsy 报告向导中使用上传与提案配置面板时的操作步骤。整个流程分为四个阶段，每个阶段对应一个 UI 组件。

### 阶段 1——进入报告向导

取证人员在 Autopsy 的报告生成向导（`Tools → Generate Report...`）中选择"Case Data Extract"作为报告模块开始操作。该模块列在"General"报告类别下。选择后，报告向导加载配置面板（`UploadSettingsPanel`）作为向导步骤序列的一部分。图 4.X 展示了选中 Case Data Extract 模块并显示配置面板的报告向导界面（此处预留截图）。

### 阶段 2——配置面板交互

配置面板（图 4.Y，此处预留截图）向取证人员呈现以下控件：

- **Gateway URL**（文本框）：Gateway 服务的 URL，例如 `http://192.168.1.100:3000`。该值一旦配置即跨会话持久化。
- **One-Time Token**（密码框）：从 gateway 管理员处获取或通过自动令牌发放机制获取的临时认证令牌。该字段在每次会话开始时均为空。
- **模式选择**（两个单选按钮）："Upload after save"用于初始提交，"Submit as modification proposal"用于发起修改提案。同一时间只能激活一种模式。
- **Proposal reason**（文本区域，仅在修改提案模式下显示）：必填自由文本字段，说明发起修改提案的原因。
- **Signing Password**（密码框，可选）：网关合约模式要求的密码，用于授权链上写入操作。
- **Collect timing metrics**（复选框）：启用后，系统记录详细的计时信息供研究评估使用。
- **Upload Receipt Path**（只读显示）：显示本地报告和上传回执的保存目录。

取证人员填写必填字段、选择期望的模式，然后继续向导操作。

### 阶段 3——本地报告生成

当取证人员点击"Generate"按钮时，模块执行 4.3.3 节描述的完整本地报告生成逻辑。进度面板显示生成状态：序列化案件元数据、加载操作日志、枚举数据源、枚举文件、计算聚合哈希。完成后，`case_data_extract.json` 写入报告输出目录（例如 `Reports/CaseDataExtract/case_data_extract.json`），并通过 `openCase.addReport(...)` 注册回 Autopsy 案件。

向导还显示一个诊断链接面板，展示生成的报告、操作日志以及（如适用）上传回执的位置。

### 阶段 4——上传或提案执行

在本地报告成功生成后，模块检查已配置的动作：

- 如果选择了 **Upload after save** 且 gateway URL 和一次性令牌有效，模块向 gateway 的上传端点（`/api/case/upload`）发送 HTTP POST 请求。请求体包含五个字段：`caseId`、`examiner`、`aggregateHash`、`generatedAt` 和 `caseJson`。取证人员可以通过向导的进度面板观察上传进度。
- 如果选择了 **Submit as modification proposal**，且提案理由字段非空，模块向 gateway 的提案端点（`/api/case/propose`）发送 HTTP POST 请求。请求体包含同样的五个字段外加提案理由。

接收到 gateway 响应后，模块将 `upload_receipt.json` 写入同一输出目录。回执（示例见图 4.Z，此处预留截图）记录以下信息：

- `indexHash` 和 `recordHash`：Gateway 返回的链上哈希值
- `txHash`：区块链交易哈希（仅合约模式）
- `blockNumber`：交易确认所在的区块号
- `requestId`：请求的唯一标识符
- `uploadStartedAt` / `uploadResponseAt`：客户端侧计时信息
- `integrityMs` / `chainMs` / `caseRegistryMs` / `totalMs`：Gateway 端各子阶段耗时（开启计时时）
- 失败时：`errorKind`、`httpStatus`、`errorMessage`

`CaseEventRecorder` 还会在操作日志中注册 `UPLOAD_SUCCESS` 或 `UPLOAD_FAILED` 事件，将上传尝试与本地报告的审计追踪关联起来。

如果上传和提案模式均未启用（两个开关都关闭），向导在本地报告生成后即完成，不发送任何 HTTP 请求。取证人员仅获得本地报告，可以在后续手动传输到外部系统。

## 三、核心设计决策及其逻辑

### 决策 1：区块链交互由独立 gateway 处理——职责分离

**设计实现：** Autopsy 插件不包含区块链节点 SDK、不执行智能合约调用、不管理密码学密钥。插件通过 `GatewayClient`（基于 `java.net.HttpURLConnection`）向独立部署的 gateway 服务发送 HTTP 请求。Gateway 负责令牌验证、哈希重计算、签名密码校验和智能合约调用。插件仅需知道 gateway URL 和一次性令牌。

**合理性：**

这是本模块最根本的架构决策，具有多层合理性。

第一，从依赖管理的角度，将区块链 SDK 直接嵌入 Autopsy 插件会引入大量第三方依赖（如 Web3j、FISCO BCOS Java SDK、加密库等）。本插件的构建环境是 Autopsy 4.22 的 NetBeans Platform，部署方式为 core JAR 补丁（见 4.3.1 节的部署分析）。在该方式下，依赖管理的灵活性非常有限——每引入一个 JAR 依赖都需要手动追加到构建脚本和补丁打包流程中。将链交互代码放在 gateway 中，使插件保持轻量，避免了在受限的部署环境中管理重型依赖。

第二，从运维灵活性的角度，gateway 更新（如升级智能合约版本、更换区块链节点、修改同步策略）可以在不触及 Autopsy 安装和插件配置的情况下独立完成。取证人员工作站的环境稳定性是数字取证的合规性要求之一——过于频繁地修改取证工具会增加验证和审计负担。

第三，从安全隔离的角度，密码学密钥（如链上账户私钥）由 gateway 管理，从不进入 Autopsy 进程。这意味着即使插件 JAR 被逆向分析，攻击者也无法提取链上凭证。这与密码学中的**密钥与使用分离**原则一致——持有凭证的组件（gateway）应当与使用凭证时操作的组件（插件）进行物理或逻辑上的隔离。

### 决策 2：初始提交与修改提案互斥——流程状态区分

**设计实现：** 配置面板上以单选按钮（radio button）方式呈现两个互斥选项："Upload after save"（初始提交）和"Submit as modification proposal"（修改提案）。在 `CaseDataExtractReportModuleSettings` 中，`uploadEnabled` 和 `proposalEnabled` 被设计为互斥的布尔值——连续调用 `setProposalEnabled(true)` 后 `setUploadEnabled` 返回 false，反之亦然。修改提案模式下，面板额外显示一个提案理由（proposal reason）文本输入框。

**合理性：**

这两种模式对应监管链中不同的程序状态。初始提交适用于**尚未在链上登记**的案件——gateway 接收到初始提交请求后，在链上创建一条新的记录，包含 `index_hash` 和 `record_hash`。修改提案适用于**已登记但需要更新**的案件——例如后期分析中发现了新的关键文件需要追加记录。在修改提案模式下，gateway 不会直接修改链上状态，而是创建一个待审批的记录，通过后续的审批流程（如副队长或另一名取证人员确认）才能最终写链。

互斥设计的内在逻辑是：一个请求不能同时是"创建"和"修改"——这在监管链的程序语义上是矛盾的。强制用户作出明确选择，避免了因配置歧义导致的操作错误（例如误将初始提交以修改提案模式发出导致被网关拒绝，或者反之）。这种设计体现了数字取证中**程序不可逆性**在软件设计层面的落实：每个操作有其明确的程序定位，跳过或混淆程序阶段可能导致证据可采性的争议。

### 决策 3：上传请求的字段设计——自包含可验证载荷

**设计实现：** 上传请求体是一个组合 JSON 对象，包含五个字段：


| 字段              | 内容                                | 目的                          |
| --------------- | --------------------------------- | --------------------------- |
| `caseId`        | 案件标识符，例如 "Case-2025-001"          | 将上传内容关联到特定的调查案件             |
| `examiner`      | 操作者姓名，例如 "Dr. Smith"              | 记录执行提交操作的人员身份               |
| `aggregateHash` | 12 字符十六进制，规范报告体的 SHA-256 截断值      | 提供报告级完整性承诺，供独立验证            |
| `generatedAt`   | ISO 8601 格式的报告生成时间戳               | 建立事件的时间线上下文                 |
| `caseJson`      | `case_data_extract.json` 的完整文本字符串 | 使 gateway 能够重算哈希，下游系统能够查询内容 |


Gateway 收到请求体后，首先对 `caseJson` 执行与 Autopsy 端相同的规范化序列化过程，重新计算聚合哈希，然后与请求中的 `aggregateHash` 字段比较。只有比对通过后才继续后续的链上操作。

**合理性：**

这五个字段并非随意选择，每个都直接满足数字取证管理中的一项要求，同时服务于本研究中区块链辅助监管链的研究主题。

`caseId` 字段确保每条链上记录唯一关联到特定的调查案件，这是任何监管链系统的前提条件（Kent et al., 2006）。没有 caseId，由它派生的链上 `indexHash` 在审计时无法链接到任何实际案件。

`examiner` 字段满足监管链文档必须记录每个阶段由谁处理证据的问责要求（Ćosić and Ćosić, 2012）。同一案件可能有多个取证人员参与（主检、协助、复核），区分提交来源对后续审计或法律程序至关重要。

`aggregateHash` 字段提供可独立验证的完整性承诺。该字段还参与 `recordHash`（链上哈希）的计算，将 gateway 端验证结果绑定到区块链承诺本身。

`generatedAt` 时间戳建立构建事件时间线所需的时间上下文（Ćosić and Ćosić, 2012），它直接支持 RQ 2.1——使研究者能够量化报告生成与链上确认之间的经过时间。

`caseJson` 字段体量最大，承担三个层面的作用，分别对应系统完整性保证的三个层次。第一，**独立验证原则**——如果仅发送哈希，gateway 将被降级为中继而非验证者。第二，**证据材料完整性**——如监管链文档标准（Ćosić and Ćosić, 2012）所要求的，`caseJson` 保留了完整的证据材料，在本地副本丢失时提供恢复来源。第三，**下游功能支持**——第 4.5 节描述的法官面板需要查询和展示完整案件内容，`caseJson` 是 gateway 提供该服务的必要前提。

发送 `caseJson` 还有一个重要的密码学后果：它支持分层完整性验证。本研究的聚合哈希不是单一哈希（一个 hash 覆盖所有内容），而是一种分层构造——每层（案件元数据、操作、文件）计算自己的子哈希，然后逐层向上聚合。如果只发送最终的聚合哈希，gateway 只能验证最终值是否一致。有了完整的报告体，gateway（以及未来的审计者）可以执行相同的规范化过程和逐层哈希过程，验证每个子层而非相信一个单一的顶层值。这种分层验证能力是 4.3.3 节聚合哈希设计的自然延伸。

### 决策 4：Session 级凭据处理——最小凭证暴露

**设计实现：** 一次性令牌（one-time token）、签名密码（signing password）和提案理由（proposal reason）这三个字段存储在 `CaseDataExtractReportModuleSettings` 对象的内存字段中，`applyTo()` 方法从设置对象读取这些字段设置 HTTP 头和请求体，但从不将它们写入 NetBeans `NbPreferences` 或任何其他持久化存储。当报告向导关闭时，这些字段随对象被垃圾回收。当 Autopsy 重启后，输入框为空，必须重新输入。Gateway URL 和开关状态（上传/提案使能选择）被持久化——但这属于配置参数而非授权凭据。

**合理性：**

这是一项遵循**最小凭证暴露**原则的设计。持久化存储中的凭据面临两个风险：一是旧凭据可能被无意中复用（例如取证人员在一次 session 中使用了一个令牌，下次打开 Autopsy 时发现字段已经填好，在没有重新生成令牌的情况下直接提交）；二是持久化存储（`NbPreferences` 在 Windows 上通常存储于注册表或 `%APPDATA%`）可能在 Autopsy 未运行时被其他进程读取。将凭据限制在内存中并且限定在 session 生命周期内，有效控制了这两个风险。

凭证不持久化还包含一个隐含的设计意图：要求取证人员每次上传时都重新获取和输入凭证。这种"每次主动输入"的制度化要求，增加了操作者对上传感知（awareness）——用户必须意识到"我正在执行一次上传操作"并为此采取了明确的凭证输入动作，而非在报告向导中无意识地点击"下一步"。

### 决策 5：本地报告生成先于上传执行——故障隔离

**设计实现：** 工作流中，4.3.3 节的本地报告生成（`generateReport()` → 写入 `case_data_extract.json` → 通过 `openCase.addReport()` 注册）始终先于上传/提案 HTTP 请求执行。即使 gateway 不可达、令牌无效或任何网络错误导致上传失败，`case_data_extract.json` 已经在磁盘上保存完好，并在 Autopsy 案件中被注册。失败信息由 `UploadReceiptWriter` 写入 `upload_receipt.json`，同时 `CaseEventRecorder.addEvent()` 在操作日志中记录 `UPLOAD_FAILED`，但本地报告本身不被动用。

**合理性：**

这种设计建立了一个明确的层级优先级：本地取证报告是**主要证据产物**（primary evidentiary artefact），链上提交是**次级完整性承诺**（secondary integrity commitment）。次要操作的失败不应影响主要产物的完整性或可访问性。

这一优先级反映了数字取证的一条基本原则：证据的存在和完整性不依赖于任何外部系统。如果本地报告的存在以成功上传为前提（例如生成报告后若上传失败则删除本地报告），那么取证人员在离线环境或网络故障时就会完全丢失证据记录——这在取证实践中是不可接受的。将两个步骤解耦，确保了本地报告的**独立存活性**——即便 gateway 永久不可用，取证人员仍然拥有一份完整的结构化取证报告和本地操作日志。

此外，`upload_receipt.json` 记录了失败详情（错误类型、HTTP 状态码、错误消息、耗时），取证人员可以据此排错后重新提交，而无需从零开始重新生成报告。这种设计体现了**事务持久性**中"不丢失已完成的局部工作"的理念。

### 决策 6：Gateway 端链前验证——防御纵深

**设计实现：** Gateway 在调用智能合约写入链之前，按顺序执行三项验证：

1. **令牌验证**：检查 HTTP `X-Auth-Token` 头中的 one-time token 是否有效且未过期
2. **聚合哈希重算**：从 `caseJson` 执行规范化序列化 → SHA-256 → 截断 12 字符，与请求中的 `aggregateHash` 比对
3. **签名密码验证**（合约模式）：调用 FISCO BCOS 合约的 `proposeUpdate` 方法前，对签名密码进行预校验

每一项验证失败都会直接返回错误响应（含 HTTP 状态码和错误消息），不会执行链上操作。

**合理性：**

这一设计不是在 Autopsy 端和 gateway 端做冗余验证，而是构建了多层次的防御纵深（Defence in Depth）。单靠 Autopsy 端的验证是不够的——如果 Autopsy 进程被篡改（例如恶意代码修改了 `generateReport()` 的输出后再计算哈希），错误的数据可能在没有外部校验的情况下被写入区块链。Gateway 端作为独立于 Autopsy 的验证节点，运行在不同的进程和（通常）不同的机器上，提供了独立的判断窗口。

令牌验证防止未经授权的第三方利用泄露的 gateway 端点进行写入操作。聚合哈希重算确保提交内容的完整性——如果 Autopsy 上的报告文件在生成后与上传前之间被静默修改（例如硬盘坏道导致文件损坏），Gateway 端重算的哈希将不匹配，请求将被拒绝，损坏的数据不会进入链上记录。签名密码验证增加了对链上写入操作的人为授权控制——只有持有签名密码的操作者才能触发区块链状态变更。

这三项验证的先后顺序也是经过考虑的：最轻量的检查（令牌验证）最先执行，最重量的是最后执行。如果令牌无效，gateway 可以立即拒绝，无需解析请求体或计算哈希。

### 决策 7：计时仪器——面向研究评估的可观测性

**设计实现：** 配置面板上有一个"Collect timing metrics"复选框。启用后，HTTP 请求包含 `X-Debug-Timing: 1` 头。Gateway 在响应中返回 `integrityMs`（聚合哈希校验耗时）、`chainMs`（CRUD 表插入耗时）、`caseRegistryMs`（智能合约执行耗时）和 `totalMs`（gateway 端总耗时）。在 Autopsy 端，`UploadClientTiming` 记录 `uploadStartedAt`、`uploadResponseAt` 和 `clientRoundTripMs`。这些时间字段写入 `upload_receipt.json`。

**合理性：**

设计中的计时功能并非用于面向用户的功能性目的，而是服务于本研究的评估需求。研究问题 RQ 2.1（延迟是否可接受）、RQ 2.2（各阶段的延迟分布）、RQ 2.3（主要瓶颈）都需要精确的时间测量数据来回答。如果系统设计中没有内置计时仪器，评估阶段只能依靠外部工具（如 tcpdump、Wireshark、手动计时）进行粗粒度测量，不仅精度不足，也无法区分 gateway 端内部各子阶段的延迟。

在取证实践中，引入区块链的工作流最常被质疑的就是延迟开销——传统的哈希验证几乎是即时的，而区块链交易需要等待共识和区块确认。内建的计时数据使研究者能够回答"每次上传延迟了多少毫秒、其中的瓶颈在哪"这一核心问题，为评价该系统在实际取证环境中的可行性提供定量依据。

## 四、整体设计的合理性总结

Upload and Proposal Configuration 模块将三个层面的考量统一在一个机制内。这三个层面呈现清晰的递进关系：首先是插件与区块链复杂性的架构隔离，其次是确保主要取证记录在操作失败时不受影响的工作流保护，最后是使上传操作在密码学上可被多方验证的验证链。

**操作边界层**：通过 gateway 委托模式（决策 1），Autopsy 插件与区块链基础设施之间的操作边界被清晰地划分开来。插件保持轻量，链交互独立演化，密钥安全管理集中到 gateway。这一边界也体现在部署维度——gateway 更新不需要重建或重新安装 Autopsy 插件。

**工作流完整性层**：通过互斥模式设计（决策 2）、本地报告优先执行（决策 5）和 Session 级凭据处理（决策 4）等决策，确保上传操作不会破坏主要证据产物的完整性，不会在持久化存储中留下可被无意复用的凭据，不会混淆不同的监管链程序阶段。这些决策共同构成了一套保护**取证工作流完整性**的机制——即便上传失败，本地记录完好；即便关闭 Autopsy，凭据不会保留到下一次 session。

**可验证性层**：自包含可验证载荷（决策 3）与链前验证检查（决策 6）共同构建了一条完整的验证链：从 **Autopsy 本地报告**，到 **gateway 验证结果**，到 **链上哈希承诺**。这条链上的每个参与者都具备独立验证能力——Autopsy 可以验证本地报告的完整性，gateway 可以验证上传内容的完整性，后续的审计者可以通过链上哈希验证报告是否被篡改。计时仪器（决策 7）虽然不是验证链本身的一部分，但提供了评估这些验证层引入的延迟在实践中是否可接受所需的经验可观测性。

这三个层面叠加的结果是：上传操作本身变得可信——不仅报告内容被传递到了 gateway，传递过程本身被验证、被记录、与被锚定到区块链的哈希形成了密码学可证明的关联。这与 ISO/IEC 27037 对数字证据获取后保护（Preservation）阶段的要求——即完整性验证应贯穿整个生命周期——高度一致。

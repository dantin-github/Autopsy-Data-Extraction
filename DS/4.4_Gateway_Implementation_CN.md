# 4.4 API Gateway 实现

## 4.4.0 整体介绍

前续章节描述了 Autopsy 端的实现——取证人员在 Autopsy 内部完成案件分析，生成本地结构化报告（4.3.3），并通过上传配置面板（4.3.5）将报告提交到网关。本节将视角转向服务端，描述承接 Autopsy 提交数据、提供查询审计服务的 API Gateway 的设计与实现。

### 网关在整个系统中的角色

API Gateway 处于 Autopsy 与下游验证基础设施（区块链）之间的中间层。它以 RESTful HTTP 服务的形式运行，承担四个核心职责：

1. **认证分流**：区分警察（证据提交者）和法官（审计者）两个角色，为它们提供不同的认证路径和权限边界
2. **上传校验**：接收 Autopsy 提交的结构化报告，在转发之前独立验证报告的完整性
3. **审批工作流管理**：当已有记录需要修改时，提供警察提案→法官审批→执行的程序化流程，防止单方面篡改
4. **查询审计**：为法官提供案件记录检索和事件历史查看接口

### 服务对象

网关服务两类客户端。第一类是 Autopsy 插件本身——它通过 `java.net.HttpURLConnection` 发送 HTTP 请求到网关，无法维护浏览器会话，因此依赖请求头中的身份令牌进行认证。第二类是法官 Web Dashboard——它是一个标准的浏览器应用，使用基于 cookie 的会话机制。这两类客户端的不同特性决定了两种认证路径的设计。

### 四个功能模块的划分

网关的实现按处理流程的四个阶段组织为四个功能模块，依次为：认证分流 → 上传完整性校验 → 两方审批工作流 → 查询审计。以下各节按此顺序展开。

---

## 4.4.1 模块一：认证分流

### 一、功能定位：解决什么核心问题

在数字取证监管链系统中，不同角色的操作者承担不同的职责。取证人员（警察）负责提取证据并提交到监管系统，而法官或检察官负责审查证据是否被篡改、检查监管链记录的完整性。这两种角色的操作范围不同，因此需要不同的认证路径和权限边界。

传统系统中常见的问题包括：所有操作者共享同一个访问凭据，无法追溯具体操作人；操作者凭据权限过大，可以进行其职责以外的操作；缺乏对操作者身份的独立验证，难以满足监管链中对"谁在什么时间做了什么"的审计要求。

本模块的设计目标是：建立双角色的认证与授权体系，使警察和法官各自由不同的认证路径进入系统，并在进入系统后仅能访问其职责范围内的功能。

### 二、核心设计决策及其逻辑

#### 决策 1：警察采用双因子认证（密码 + OTP）

警察的认证路径包含两个因子。第一因子是密码验证——用户的密码以 bcrypt 哈希形式存储在本地用户文件中，网关通过 `bcrypt.compare()` 进行比对。第二因子是一次性密码（OTP）——密码验证通过后，网关生成一个 16 字节随机十六进制字符串作为临时令牌，通过邮件发送到操作者的邮箱。

**合理性分析：**

这一设计解决了两个层面的问题。首先，双因子认证符合数字取证监管链中对操作者身份验证的严格性要求。仅凭密码不足以建立与特定操作者之间的高置信度绑定——密码可能被分享、猜测或记录在便签上。邮件的额外验证步骤确保操作者确实能够访问其声称的工作邮箱，增加了身份验证的强度。

其次，OTP 的临时性比长期的 API Key 更适合取证操作的审计要求。每一次上传操作都需要获取一个新的 OTP，这意味着每一次证据提交都对应一次独立的授权事件。如果使用长期 API Key，一个被泄露的 Key 可以在不被察觉的情况下被多次使用。

在设计上，OTP 支持两种消耗模式：单次使用模式（默认方式，OTP 在首次使用后即失效，可防止重放攻击）和可复用模式（OTP 在其有效期内可多次使用，适用于批量上传多个案件的场景）。

#### 决策 2：法官采用密码加会话的直接认证

法官的认证路径更为简洁——通过密码验证后直接创建浏览器会话，无需 OTP。

**合理性分析：**

这一差异化的设计基于两种角色的操作模式不同。警察的操作通常发生在现场——他们从 Autopsy 工作站直接上传证据报告，可能无法始终通过浏览器访问 Web 界面。因此，警察需要令牌方式以便让 Autopsy 插件代其提交。而法官的操作完全通过 Web Dashboard 进行，他们坐在办公室中使用浏览器审阅证据，因此基于 cookie 的会话机制已经足够。法官的工作场景也不像警察那样有高频批量上传需求，无需引入临时的 OTP 流程增加操作摩擦。

#### 决策 3：令牌认证与会话认证的双通道分离

网关为警察和法官分别提供不同的认证通道。警察的 Autopsy 插件通过 HTTP 头 `X-Auth-Token` 提交 OTP 进行认证，适用于程序化请求。警察的 Web 会话可以通过一个专门端点将 OTP 兑换为会话 cookie，适用于浏览器操作。法官则完全通过标准 Web 登录页面创建会话。

**合理性分析：**

这一分离的核心驱动力是 Autopsy 插件与 Web 应用在 HTTP 通信层面的本质差异。Autopsy 插件使用 `java.net.HttpURLConnection` 发送 HTTP 请求，这种客户端无法存储和管理浏览器 cookie，也无法处理基于 cookie 的会话续期逻辑。如果强制要求插件维护会话状态，就需要在 Autopsy 端引入额外的 HTTP 会话管理代码，增加了插件的复杂度。令牌方案使插件只需在每个请求中附加一个固定的头字段即可完成认证，最为简洁。

而对于 Web 应用场景，标准 `express-session` cookie 提供了更自然的用户体验——法官登录一次后，在其会话有效期内不必重复输入凭据。两者各司其职。

### 三、操作流程概述

警察上传证据时的认证流程如下：操作者在 Autopsy 报告配置面板中输入网关地址和 OTP，插件携带该令牌发送上传请求；网关验证令牌的有效性和过期时间，通过后将请求路由到上传处理逻辑。如果操作者需要从浏览器发起修改提案，则先通过 Web 登录页输入密码和 OTP，将 OTP 兑换为浏览器会话后开始操作。

法官的认证流程更为直接：打开 Web Dashboard 登录页面，输入密码，服务器创建会话后重定向到 Dashboard 主界面。法官在其会话期间可以查看案件、审批或驳回修改提案。

两个角色的权限边界由网关在路由层面保证——警察认证通道无法访问法官的查询和审批端点，法官会话也无法访问上传和提案端点。

## 4.4.2 模块二：上传完整性校验

### 一、功能定位：解决什么核心问题

4.3.3 节生成的本地报告（`case_data_extract.json`）包含了完整的案件信息和聚合哈希值。这份报告从 Autopsy 工作站传输到网关的过程中，面临几个风险：传输过程中的数据损坏可能导致网关收到不完整的或损坏的报告内容；如果仅传递哈希值而不传递完整报告内容，网关将没有能力进行任何独立验证，只能盲目信任客户端声称的值。

本模块的设计目标是：在 Autopsy 的提交数据到达网关时，对报告内容的内部一致性进行独立校验——即验证接收到的报告内容与接收到的聚合哈希值是否匹配。通过这一校验，网关确保进入系统的数据是自洽的；任何导致内容与哈希不一致的传输错误或恶意修改，都会被校验机制拒绝。

### 二、操作流程概述

整个上传操作从开始到完成的流程如下：

警察操作者在 Autopsy 中完成报告生成后，在配置面板中输入网关 URL 和从管理员处获取的 OTP。Autopsy 插件将包含 `caseId`、`examiner`、`aggregateHash`、`generatedAt` 和完整报告内容的 JSON 请求体发送到网关的上传端点，并在 HTTP 头中携带 OTP。

网关接收到请求后，首先验证身份令牌——确认令牌存在、未过期且未被使用过（如果处于单次使用模式）。令牌验证通过后，网关检查请求体中是否包含所有必填字段，以及该 `caseId` 是否已经被注册过。

字段验证通过后，网关从 `caseJson` 中提取报告内容，使用与 Autopsy 端一致的规范 JSON 规则重新计算聚合哈希，并与提交的 `aggregateHash` 进行比较。如果哈希不匹配，网关立即返回错误，请求不会被进一步处理。

哈希验证通过后，网关计算两个密码学标识符——`indexHash`（由案件 ID 的 SHA-256 得到，作为链上查找的键）和 `recordHash`（由案件 ID、操作者姓名、聚合哈希、生成时间戳和报告内容的拼接字符串的 SHA-256 得到，作为本案提交记录的密码学承诺）。随后，网关将 `indexHash` 和 `recordHash` 写入区块链（通过 CRUD 表或智能合约），并将报告内容更新到本地记录存储。最后，网关向上传客户端返回响应，包含 `indexHash`、`recordHash`、写入确认信息以及可选的计时数据。

如果启用了计时功能，网关还会在响应中返回各阶段的耗时数据（`integrityMs`、`chainMs`、`totalMs` 等），供研究评估使用。这些计时字段默认不启用，仅在请求头中包含 `X-Debug-Timing: 1` 时才会出现在响应中。

### 三、核心设计决策及其逻辑

#### 决策 1：上传内容包含完整报告而非仅哈希值

网关在接受上传请求时，要求客户端（Autopsy 插件）提交 `case_data_extract.json` 的完整文本内容（`caseJson` 字段），而非仅提交一个哈希值。网关从该完整内容中独立重新计算聚合哈希，并与客户端提交的 `aggregateHash` 进行比较。

**合理性分析：**

这是本模块最核心的设计决策。如果 Autopsy 只提交哈希值而不提交报告全文，网关将没有能力进行任何独立的验证——它只能将接收到的哈希值原样写入区块链，完全信任客户端提供的数据。提交完整报告使网关获得了对数据进行独立检查的能力：它可以重新计算哈希并与提交值比对，验证数据在到达网关时至少是内部自洽的。

这一设计也支持后续的监管链审计：即使链上仅保留哈希承诺，审计者也可以通过网关本地保留的报告内容随时重新计算哈希，与链上记录的哈希进行比对，而不需要依赖 Autopsy 端的任何数据。

#### 决策 2：本地记录存储保留完整报告内容用于独立验证与权威记录

网关在本地维护一个记录存储（`~/.case_record_store.json`），用于保存每次成功上传的完整报告内容。该存储不是区块链存储的缓存——它保存了完整的 `caseJson` 文本，而链上仅保留哈希承诺。当法官通过查询接口检索案件时，网关首先从本地存储中返回报告全文，同时从区块链上获取 `recordHash`，然后网关自行重新计算报告内容的哈希并与链上记录比对，在响应中同时返回报告内容和比对结果。

**合理性分析：**

本地记录存储的意义从两个层面来理解。

第一，链上链下的查询对应关系。区块链通常只保存哈希值而非完整数据，这是由区块链的存储成本决定的。如果网关也不保留完整报告，那么当需要审计时，查询者将无法获取报告原文——只能看到哈希值，而哈希值本身不包含任何案件信息，无法用于实质审查。通过将完整报告保留在网关本地、将哈希承诺写入不可篡改的链上存储，审计者可以用本地存储快速检索和查看报告内容，再用链上哈希验证内容是否被篡改——查询效率与安全性各得其所。

第二，网关作为报告存储的权威机构。在现实的法律管控体系中，证据的流转通常经过一个集中的权威节点——例如公安机关的电子数据取证实验室或物证管理中心。这个节点负责接收和保管各基层单位提交的证据材料，对材料的接收时间、来源、完整性进行登记，并作为后续司法程序中提供参考的"官方版本"。网关在本系统中扮演了类似的角色。

网关记录的报告版本相较于 Autopsy 工作站的本地文件具有更强的权威性，原因有三。其一，Autopsy 工作站是分布式操作——多个警察可以在各自的机器上进行独立分析，每个工作站上生成的报告是操作者的个人工作产物，未经统一核验。而所有提交到网关的报告都经过了一次独立的完整性校验，只有通过校验的数据才被接纳。其二，Autopsy 工作站的本地文件可以被操作者随意修改或删除——Autopsy 本身没有对已生成的报告文件施加写保护。而网关在接收报告后，其内容不再允许被上传者单方面修改（需要通过审批工作流），版本的稳定性远高于工作站本地副本。其三，网关记录与链上哈希承诺形成一一对应——每份报告的内容都可以通过链上哈希独立验证，而 Autopsy 工作站的本地文件缺乏这种链上锚定。因此，网关保存的报告版本承担了"可信副本"的角色——当调查需要引用一份报告时，应当以网关中的记录为准，而非原始工作站上的文件。

#### 决策 3：聚合哈希重算使用与 Autopsy 相同的规范 JSON 规则

网关端的聚合哈希重算必须使用与 Autopsy 端完全相同的规范 JSON 序列化规则，以确保两者对相同输入产生完全相同的输出。该规则包括：解析 JSON 结构体、移除四个与完整性承诺无关的元数据字段、按字典序递归排序所有对象键、无多余空格或换行符序列化、计算 SHA-256、取前 12 个十六进制字符。

**合理性分析：**

跨语言哈希一致性是这一设计的密码学基础。如果网关重算时使用的 JSON 序列化规则与 Autopsy 有任何差异（例如空格处理方式不同、键排序顺序不同），即使报告内容完全相同，两者计算出的哈希也会不同，导致网关错误地拒绝合法请求。由于 Autopsy 端使用 Java 实现而网关使用 Node.js/JavaScript 实现，两种语言的标准 JSON 库的行为存在差异——例如 Java 的 `HashMap` 不保证键的迭代顺序，而 JavaScript 的 `JSON.stringify` 从 ES2015 开始保证对象键的插入顺序。因此，规范 JSON 序列化规则必须在两种语言中**独立实现**，并通过共享测试夹具进行交叉验证。

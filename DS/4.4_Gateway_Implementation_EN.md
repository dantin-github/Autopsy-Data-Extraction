# 4.4 API Gateway Implementation

## 4.4.0 Overview

The preceding sections described the Autopsy-side implementation — how an examiner completes case analysis within Autopsy, generates a local structured report (Section 4.3.3), and submits the report to the gateway through the upload configuration panel (Section 4.3.5). This section shifts the perspective to the server side, describing the design and implementation of the API Gateway that receives submissions from Autopsy and provides query and audit services.

### Role of the Gateway in the Overall System

The API Gateway sits as a middle layer between Autopsy and the downstream verification infrastructure (blockchain). It runs as a RESTful HTTP service and assumes four core responsibilities:

1. **Authentication and role routing**: Distinguishes between two operator roles — police (evidence submitter) and judge (auditor) — and provides distinct authentication paths and permission boundaries for each.
2. **Upload and integrity verification**: Receives the structured report from Autopsy and independently verifies its integrity before forwarding.
3. **Approval workflow management**: Provides a procedural path — police proposal, judge review, and execution — for modifying existing records, preventing unilateral tampering.
4. **Query and audit**: Provides case record retrieval and event history inspection interfaces for judges.

### Clients Served

The gateway serves two types of clients. The first is the Autopsy plugin itself — it sends HTTP requests via `java.net.HttpURLConnection`, which cannot maintain browser sessions, so it relies on identity tokens carried in request headers. The second is the judge Web Dashboard — a standard browser application that uses cookie-based sessions. The differing characteristics of these two client types drive the design of two distinct authentication paths.

### Four Functional Modules

The gateway implementation is organised into four functional modules corresponding to four sequential stages of the processing pipeline: Authentication and Role Routing → Upload and Integrity Verification → Two-Party Approval Workflow → Query and Audit. The following subsections describe each module in this order.

---

## 4.4.1 Module 1: Authentication and Role Routing

### 1. Functional Positioning: What Core Problem Does It Solve?

In a digital forensic chain-of-custody system, different operator roles bear different responsibilities. The examiner (police) is responsible for extracting evidence and submitting it to the custody system, while the judge or prosecutor is responsible for reviewing whether the evidence has been tampered with and checking the integrity of the chain-of-custody records. These two roles require different authentication paths and permission boundaries.

Common problems in traditional systems include: all operators share the same access credential, making it impossible to trace who performed which operation; operator credentials grant excessive permissions, allowing actions beyond the operator's duties; and a lack of independent identity verification for operators, making it difficult to meet the chain-of-custody requirement of recording "who did what and when."

The design goal of this module is to establish a dual-role authentication and authorisation system. Police and judges each enter the system through different authentication paths and, once authenticated, can only access functions within their respective scope of duties.

### 2. Core Design Decisions and Their Rationale

#### Decision 1: Two-Factor Authentication for Police (Password + OTP)

The police authentication path consists of two factors. The first factor is password verification — user passwords are stored as bcrypt hashes in a local user file, and the gateway compares them using `bcrypt.compare()`. The second factor is a one-time password (OTP) — after password verification succeeds, the gateway generates a 16-byte random hex string as a temporary token and delivers it to the operator's email via `nodemailer`.

**Rationale:**

This design addresses two problems. First, two-factor authentication meets the strictness requirements for operator identity verification in digital forensic chain-of-custody practices. A password alone is insufficient to establish a high-confidence binding to a specific operator — passwords can be shared, guessed, or written on sticky notes. The additional email verification step ensures the operator has access to their claimed work email, increasing the strength of identity verification.

Second, the temporary nature of OTPs is better suited to the audit requirements of forensic operations than long-lived API keys. Each upload operation requires obtaining a new OTP, meaning each evidence submission corresponds to an independent authorisation event. If a long-lived API key were used instead, a compromised key could be reused multiple times without detection.

The OTP supports two consumption modes: single-use mode (default — the OTP is invalidated after first use, preventing replay attacks) and reusable mode (the OTP can be used multiple times within its TTL, supporting batch upload scenarios).

#### Decision 2: Direct Password-and-Session Authentication for Judges

The judge authentication path is simpler — password verification followed immediately by session creation, without an OTP step.

**Rationale:**

This differential design is based on the different operating models of the two roles. Police typically work in the field — they upload evidence reports directly from Autopsy workstations and may not always have browser access to a web interface. Therefore, police need a token mechanism so that the Autopsy plugin can submit on their behalf. Judges, by contrast, operate entirely through the Web Dashboard — they review evidence from their office using a browser, so cookie-based sessions are sufficient. Judges also do not have the same high-frequency batch upload requirements as police, so introducing an OTP step would add unnecessary operational friction.

#### Decision 3: Dual-Channel Separation of Token Authentication and Session Authentication

The gateway provides different authentication channels for police and judges. The police Autopsy plugin submits the OTP through the `X-Auth-Token` HTTP header for programmatic requests. A police web session can be obtained by exchanging the OTP for a session cookie through a dedicated endpoint, supporting browser-based operations. Judges authenticate entirely through a standard web login page that creates a session.

**Rationale:**

The core driver for this separation is the fundamental difference between how the Autopsy plugin and a web application handle HTTP communication. The Autopsy plugin uses `java.net.HttpURLConnection` to send HTTP requests — this type of client cannot store or manage browser cookies, nor can it handle cookie-based session renewal logic. Forcing the plugin to maintain session state would require additional HTTP session management code on the Autopsy side, increasing plugin complexity. The token approach keeps things simple — the plugin only needs to attach a fixed header field to each request.

For the web application scenario, standard `express-session` cookies provide a more natural user experience — once a judge logs in, they do not need to re-enter credentials for the duration of the session. Each mechanism serves its purpose.

### 3. Operational Flow Overview

The authentication flow for police uploading evidence proceeds as follows: the operator enters the gateway URL and OTP in the Autopsy report configuration panel; the plugin carries this token when sending upload requests; the gateway verifies the token's validity and expiration, then routes the request to the upload processing logic. If the operator needs to initiate a modification proposal from a browser, they first log in through the web login page with their password and OTP, exchange the OTP for a browser session, and then begin operations.

The authentication flow for judges is more direct: open the Web Dashboard login page, enter the password, the server creates a session, and redirects to the Dashboard main interface. During the session, judges can view cases, approve or reject modification proposals.

The permission boundaries between the two roles are enforced at the routing layer — the police authentication channel cannot access the judge's query and approval endpoints, and the judge session cannot access the upload and proposal endpoints.

## 4.4.2 Module 2: Upload and Integrity Verification

### 1. Functional Positioning: What Core Problem Does It Solve?

The local report (`case_data_extract.json`) generated in Section 4.3.3 contains complete case information and an aggregate hash. During transmission from the Autopsy workstation to the gateway, several risks exist: data corruption may cause the gateway to receive incomplete or corrupted report content; if only the hash value were transmitted without the full report content, the gateway would have no capability to perform any independent verification and would simply trust the client-declared value.

The design goal of this module is to perform an independent internal consistency check when data from Autopsy arrives at the gateway — verifying that the received report content matches the received aggregate hash value. Through this check, the gateway ensures that the data entering the system is self-consistent; any transmission error or malicious modification that causes a mismatch between content and hash will be rejected by the verification mechanism.

### 2. Operational Flow Overview

The complete upload operation from start to finish proceeds as follows:

After completing report generation in Autopsy, the police operator enters the gateway URL and the OTP obtained from the administrator in the configuration panel. The Autopsy plugin sends a JSON request body containing `caseId`, `examiner`, `aggregateHash`, `generatedAt`, and the full report content to the gateway's upload endpoint, carrying the OTP in an HTTP header.

Upon receiving the request, the gateway first validates the identity token — confirming that the token exists, has not expired, and has not been consumed (if in single-use mode). After token validation passes, the gateway checks whether all required fields are present in the request body and whether the case ID has already been registered.

After field validation, the gateway extracts the report content from `caseJson`, recomputes the aggregate hash using the same canonical JSON rules as the Autopsy side, and compares it with the submitted `aggregateHash`. If the hashes do not match, the gateway immediately returns an error, and the request is not processed further.

After hash verification passes, the gateway computes two cryptographic identifiers — `indexHash` (SHA-256 of the case ID, serving as the lookup key on-chain) and `recordHash` (SHA-256 of the concatenation of case ID, examiner name, aggregate hash, generation timestamp, and report content, serving as the cryptographic commitment for this submission). The gateway then writes `indexHash` and `recordHash` to the blockchain (via CRUD table or smart contract) and updates the local record store with the report content. Finally, the gateway returns a response containing `indexHash`, `recordHash`, write confirmation information, and optional timing data.

If timing is enabled, the gateway also returns stage-level latency data (e.g., `integrityMs`, `chainMs`, `totalMs`) in the response for research evaluation purposes. These timing fields are disabled by default and appear in the response only when the request header `X-Debug-Timing: 1` is present.

### 3. Core Design Decisions and Their Rationale

#### Decision 1: Upload Content Includes the Full Report, Not Just the Hash

When accepting an upload request, the gateway requires the client (Autopsy plugin) to submit the full text of `case_data_extract.json` (the `caseJson` field), rather than only a hash value. The gateway independently recomputes the aggregate hash from this full content and compares it with the submitted `aggregateHash`.

**Rationale:**

This is the most critical design decision in this module. If Autopsy submitted only a hash value without the full report text, the gateway would have no capability to perform any independent verification — it could only write the received hash directly onto the blockchain, fully trusting the client-provided data. Submitting the full report gives the gateway the ability to independently check the data: it can recompute the hash and compare it with the submitted value, verifying that the data is at least internally self-consistent when it arrives at the gateway.

This design also supports subsequent chain-of-custody auditing: even if only the hash commitment is stored on-chain, an auditor can always recompute the hash from the report content stored locally on the gateway and compare it with the on-chain record, without needing any data from the Autopsy side.

#### Decision 2: Local Record Store Retains Full Report Content for Independent Verification and Authoritative Record

The gateway maintains a local record store (`~/.case_record_store.json`) that preserves the complete report content for each successful upload. This store is not a cache of the blockchain — it holds the full `caseJson` text, whereas the blockchain retains only hash commitments. When a judge retrieves a case through the query interface, the gateway returns the full report text from local storage while simultaneously fetching the `recordHash` from the blockchain. The gateway then independently recomputes the hash of the report content, compares it with the on-chain record, and includes both the report content and the comparison result in the response.

**Rationale:**

The significance of the local record store is understood from two perspectives.

First, the query correspondence between on-chain and off-chain storage. The blockchain typically retains only hash values rather than full data, driven by its storage cost. Without a local copy on the gateway, an auditor would be unable to retrieve the original report text — they could only see a hash value, which carries no case information and cannot support substantive review. By retaining the full report on the gateway while writing hash commitments to immutable on-chain storage, the auditor can quickly retrieve and review report content through local storage, and then use the on-chain hash to verify whether the content has been tampered with — query efficiency and security are each served by their respective storage layer.

Second, the gateway serves as an authoritative repository for report storage. In a real-world legal governance system, the chain of custody typically passes through a centralised authoritative node — such as a digital forensics laboratory or evidence management centre operated by law enforcement. This node is responsible for receiving and safekeeping evidence submitted by various基层 units, registering the time of receipt, source, and integrity of each item, and serving as the "official version" for subsequent judicial proceedings. The gateway plays an analogous role in this system.

The version of the report stored on the gateway carries stronger authority than the local files on an Autopsy workstation for three reasons. First, Autopsy workstations operate in a distributed manner — multiple police officers conduct independent analyses on their own machines, and the reports generated on each workstation are individual work products that have not undergone unified verification. In contrast, every report submitted to the gateway has passed an independent integrity check, and only data that passes the check is accepted. Second, local files on an Autopsy workstation can be freely modified or deleted by the operator — Autopsy itself does not impose write protection on generated report files. On the gateway, once a report is received, its content cannot be unilaterally modified by the submitter (modification requires going through the approval workflow), making version stability far higher than that of a workstation local copy. Third, each gateway record corresponds one-to-one with an on-chain hash commitment — the content of every report can be independently verified through its on-chain hash, whereas local files on an Autopsy workstation lack this on-chain anchoring. Therefore, the report version stored on the gateway assumes the role of "trusted copy" — when an investigation needs to cite a report, the record on the gateway should prevail over the original file on the workstation.

#### Decision 3: Aggregate Hash Recomputation Uses the Same Canonical JSON Rules as Autopsy

The aggregate hash recomputation on the gateway side must use exactly the same canonical JSON serialisation rules as the Autopsy side, ensuring that both produce identical output for the same input. The rules include: parse the JSON structure, remove the four metadata fields unrelated to the integrity commitment, recursively sort all object keys lexicographically, serialise without extra whitespace or line breaks, compute SHA-256, and take the first 12 hex characters.

**Rationale:**

Cross-language hash consistency is the cryptographic foundation of this design. If the JSON serialisation rules used by the gateway during recomputation differ in any way from those used by Autopsy (e.g., different whitespace handling, different key ordering), even if the report content is identical, the two computed hashes would differ, causing the gateway to incorrectly reject a legitimate request. Since the Autopsy side is implemented in Java and the gateway in Node.js/JavaScript, the standard JSON libraries in the two languages behave differently — for instance, Java's `HashMap` does not guarantee key iteration order, while JavaScript's `JSON.stringify` has guaranteed key insertion order since ES2015. Therefore, the canonical JSON serialisation rules must be **independently implemented** in both languages and validated through shared test fixtures.

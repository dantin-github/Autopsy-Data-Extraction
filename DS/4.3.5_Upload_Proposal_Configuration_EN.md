# 4.3.5 Upload and Proposal Configuration — Design Logic and Rationality Analysis

## 1. Functional Positioning: What Core Problem is Solved

The structured report generation module described in Section 4.3.3 can produce a complete forensic record (`case_data_extract.json`) within Autopsy, containing case metadata, an operation log, data source information, a file inventory, and the aggregate hash. However, this report remains on the examiner's local workstation after generation. To bring it into the blockchain-assisted chain-of-custody workflow — that is, to anchor its integrity commitment to the blockchain — a data channel from Autopsy to the gateway and then to the blockchain must be established.

The traditional approach suffers from a clear deficiency: examiners must manually copy the report file from the Autopsy directory, upload it to a backend system through a separate client, and manually record the upload outcome. This process separates forensic analysis from integrity anchoring — the generation time, upload time, and on-chain confirmation time lack automatic correlation; the audit trail depends on manual discipline; and there is no mechanism to prevent the situation where the local report is generated but the examiner forgets to upload it.

The design goal of this module is to embed a configuration layer inside the Autopsy report wizard, so that after the local report is generated, the examiner can choose through the same interface to either submit the report to the gateway (initial submission) or propose a modification to an already registered record (modification proposal), and have the upload result automatically recorded in a local receipt. The core intent is to make the upload operation a natural extension of the report generation workflow, rather than an external operation separated from the forensic tool.

The relationship with the report generation module in Section 4.3.3 must be clarified here. The product of Section 4.3.3 is the local structured report file — the primary evidentiary record on the examiner's workstation. The product described in this section is an HTTP request body that wraps that local report as its `caseJson` field, together with additional submission-level metadata fields. The two are not independent report formats; they are related as **source to submitted copy** — local report generation must execute before the upload, and the upload content is a packaged submission form of the local report.

## 2. Module Operational Mechanism

This section describes the operational steps that an examiner follows when using the upload and proposal configuration panel within the Autopsy report wizard. The flow consists of four sequential stages, each with a corresponding UI component.

### Stage 1 — Report Wizard Entry

The examiner initiates the flow by selecting "Case Data Extract" as the report module in Autopsy's report generation wizard (`Tools → Generate Report...`). The module is listed under the "General" report category. Upon selection, the report wizard loads the configuration panel (`UploadSettingsPanel`) as part of the wizard's step sequence. Figure 4.X shows the report wizard with the Case Data Extract module selected and the configuration panel displayed.

### Stage 2 — Configuration Panel Interaction

The configuration panel (Figure 4.Y) presents several controls to the examiner:

- **Gateway URL** (text input): The URL of the gateway service, e.g., `http://192.168.1.100:3000`. This value is persisted across sessions once configured.
- **One-Time Token** (password input): A temporary authentication token obtained from the gateway administrator or an automated token issuance mechanism. This field is always empty at the start of a session.
- **Mode selection** (two radio buttons): "Upload after save" for initial submission, or "Submit as modification proposal" for proposing changes. Only one mode can be active at a time.
- **Proposal reason** (text area, shown only when modification proposal mode is selected): A mandatory free-text field explaining the reason for the proposed modification.
- **Signing Password** (password input, optional): A password required by the gateway's contract mode for authorising on-chain write operations.
- **Collect timing metrics** (checkbox): When enabled, the system records detailed timing information for research evaluation purposes.
- **Upload Receipt Path** (read-only display): Shows the directory where the local report and upload receipt will be saved.

The examiner fills in the required fields, selects the desired mode, and proceeds through the wizard.

### Stage 3 — Local Report Generation

When the examiner clicks the "Generate" button, the module executes the complete local report generation logic described in Section 4.3.3. The progress panel shows the generation status: serialising case metadata, loading the operation log, enumerating data sources, enumerating files, computing the aggregate hash. Upon completion, `case_data_extract.json` is written to the report output directory (e.g., `Reports/CaseDataExtract/case_data_extract.json`) and registered back into the Autopsy case via `openCase.addReport(...)`.

The wizard also displays a diagnostic link panel showing the locations of the generated report, the operation log, and (if applicable) the upload receipt.

### Stage 4 — Upload or Proposal Execution

After the local report is successfully generated, the module checks the configured action:

- If **Upload after save** was selected and the gateway URL and one-time token are valid, the module sends an HTTP POST request to the gateway's upload endpoint (`/api/case/upload`). The request body contains five fields: `caseId`, `examiner`, `aggregateHash`, `generatedAt`, and `caseJson`. The examiner can observe the upload progress through the wizard's progress panel.

- If **Submit as modification proposal** was selected, and additionally the proposal reason field is non-empty, the module sends an HTTP POST request to the gateway's proposal endpoint (`/api/case/propose`). The request body includes the same five fields plus the proposal reason.

Upon receiving the gateway response, the module writes `upload_receipt.json` to the same output directory. The receipt (example shown in Figure 4.Z) records:

- `indexHash` and `recordHash`: The on-chain hashes returned by the gateway
- `txHash`: The blockchain transaction hash (contract mode only)
- `blockNumber`: The block number in which the transaction was confirmed
- `requestId`: A unique identifier for the request
- `uploadStartedAt` / `uploadResponseAt`: Client-side timing information
- `integrityMs` / `chainMs` / `caseRegistryMs` / `totalMs`: Gateway-side timing breakdown (when timing is enabled)
- For failures: `errorKind`, `httpStatus`, `errorMessage`

The `CaseEventRecorder` also registers an `UPLOAD_SUCCESS` or `UPLOAD_FAILED` event in the operation log, linking the upload attempt to the local report's audit trail.

If neither upload nor proposal mode is enabled (both toggles are off), the wizard completes after local report generation without sending any HTTP request. The examiner obtains only the local report and can manually transfer it to external systems later.

## 3. Core Design Decisions and Their Logic

### Decision 1: Blockchain Interaction Handled by a Separate Gateway — Separation of Responsibilities

**Design implementation:** The Autopsy plugin does not contain a blockchain node SDK, does not invoke smart contracts, and does not manage cryptographic keys. The plugin communicates with an independently deployed gateway service through `GatewayClient` (based on `java.net.HttpURLConnection`). The gateway is responsible for token verification, hash recomputation, signing password validation, and smart-contract invocation. The plugin only needs to know the gateway URL and a one-time token.

**Rationality:**

This is the most fundamental architectural decision of this module and has multiple layers of justification.

First, from a dependency management perspective, embedding a blockchain SDK directly into the Autopsy plugin would introduce numerous third-party dependencies (e.g., Web3j, FISCO BCOS Java SDK, cryptographic libraries). The build environment for this plugin is Autopsy 4.22's NetBeans Platform, deployed via the core JAR patch mechanism described in Section 4.3.1. Under this deployment model, dependency management flexibility is very limited — each additional JAR dependency must be manually appended to the build script and the patching pipeline. Placing the chain interaction code in the gateway keeps the plugin lightweight and avoids managing heavy dependencies in a constrained deployment environment.

Second, from an operational flexibility perspective, gateway updates (e.g., upgrading the smart-contract version, replacing blockchain nodes, modifying the synchronisation strategy) can be carried out independently without touching the Autopsy installation or plugin configuration. The stability of the examiner's workstation environment is a compliance requirement in digital forensics — modifying forensic tools too frequently increases the burden of verification and audit.

Third, from a security isolation perspective, cryptographic keys (such as the on-chain account private key) are managed by the gateway and never enter the Autopsy process. This means that even if the plugin JAR is reverse-engineered, an attacker cannot extract the on-chain credentials. This is consistent with the principle of **key-operation separation** in cryptography — the component holding the credentials (gateway) should be physically or logically isolated from the component that uses the operation result (plugin).

### Decision 2: Initial Submission and Modification Proposal are Mutually Exclusive — Procedural State Distinction

**Design implementation:** The configuration panel presents two mutually exclusive options as radio buttons: "Upload after save" (initial submission) and "Submit as modification proposal" (modification proposal). In `CaseDataExtractReportModuleSettings`, `uploadEnabled` and `proposalEnabled` are designed as mutually exclusive boolean values — calling `setProposalEnabled(true)` causes subsequent `setUploadEnabled` to return `false`, and vice versa. In modification proposal mode, the panel additionally displays a proposal reason text input field.

**Rationality:**

These two modes correspond to different procedural states in the chain of custody. Initial submission is for cases that have **not yet been registered** on the blockchain — upon receiving an initial submission request, the gateway creates a new on-chain record with `index_hash` and `record_hash`. Modification proposal is for cases that **have been registered but need an update** — for example, when new critical files are discovered during later analysis and need to be appended to the record. In modification proposal mode, the gateway does not directly modify the on-chain state; instead, it creates a pending record that can only be written to the chain after passing through an approval workflow (e.g., confirmation by a deputy team leader or another examiner).

The logic behind mutual exclusivity is that a single request cannot simultaneously be a "create" and a "modify" operation — this is semantically contradictory in chain-of-custody procedure. Forcing an explicit choice prevents operational errors caused by configuration ambiguity (e.g., accidentally sending an initial submission as a modification proposal and being rejected, or vice versa). This design reflects the implementation at the software level of the principle of **procedural irreversibility** in digital forensics: each operation has a clear procedural position, and skipping or confusing procedural stages could compromise evidence admissibility.

### Decision 3: Upload Content Includes the Complete Local Report — Self-Contained Verification

**Design implementation:** The upload request body is a composite JSON object with five fields:

| Field | Content | Purpose |
|---|---|---|
| `caseId` | Case identifier, e.g., "Case-2025-001" | Associates the submission with a specific investigation |
| `examiner` | Operator name, e.g., "Dr. Smith" | Records who performed the submission |
| `aggregateHash` | 12-char hex, truncated SHA-256 of canonical report | Provides a report-level integrity commitment for independent verification |
| `generatedAt` | ISO 8601 timestamp of report generation | Establishes temporal context for the chain of events |
| `caseJson` | Full text of `case_data_extract.json` as a string | Enables gateway to recompute hash and downstream systems to query content |

The gateway, upon receiving this body, first recomputes the aggregate hash from `caseJson` using the same canonical serialisation procedure, and compares the result with the `aggregateHash` field. Only after this comparison passes does it proceed with subsequent chain operations.

**Rationality:**

The five fields are not arbitrary. Each directly fulfills a digital forensics management requirement while serving the research theme of blockchain-assisted chain of custody.

The `caseId` field ensures that every on-chain record is uniquely associated with a specific investigation, a prerequisite for any chain-of-custody system (Kent et al., 2006). Without it, the on-chain `indexHash` derived from `caseId` could not be linked back to any real case during audit.

The `examiner` field satisfies the accountability requirement that chain-of-custody documentation must record who handled the evidence at each stage (Ćosić and Ćosić, 2012). Multiple examiners may be involved in the same case (lead analyst, assistant, reviewer), and distinguishing the submission source is essential for later audit or legal proceedings.

The `aggregateHash` field provides an independently verifiable integrity commitment. The same field also participates in the computation of `recordHash` — the on-chain hash — binding the gateway-side verification result into the blockchain commitment itself.

The `generatedAt` timestamp establishes the temporal context required for constructing a timeline of events (Ćosić and Ćosić, 2012). It also directly supports RQ 2.1 by enabling measurement of the elapsed time between report generation and on-chain confirmation.

The `caseJson` field — the most substantial — serves three purposes that correspond to three layers of the system's integrity guarantee. First, it enables the **independent verification principle** to operate at the gateway: if only a hash were sent, the gateway would be a relay rather than a verifier. Second, it preserves **complete evidentiary material** as required by chain-of-custody documentation standards (Ćosić and Ćosić, 2012), providing a recovery source if the local copy is lost. Third, it supports the **downstream judge panel** described in Section 4.5, which needs to query and display the full case content. In each case, the inclusion of the full report body transforms the gateway from a passive forwarding node into an active verification participant.

An important consequence of sending `caseJson` is that it enables layered integrity verification. The aggregate hash in this study is not a single hash covering everything; it is a layered construction — each layer (case metadata, operations, files) computes its own sub-hash, which is then aggregated upward. If only the final aggregate hash were sent, the gateway could only verify that the final value matches. With the full report body, the gateway (and future auditors) can execute the same canonicalisation and layer-by-layer hashing procedure, verifying the integrity of each sub-layer rather than trusting a single top-level value. This layered capability is a natural extension of the aggregate hash design from Section 4.3.3.

### Decision 4: Session-Level Credential Handling — Minimal Credential Exposure

**Design implementation:** Three fields — the one-time token, signing password, and proposal reason — are stored in memory fields of the `CaseDataExtractReportModuleSettings` object. The `applyTo()` method reads these fields from the settings object to set the HTTP header and request body, but never writes them to NetBeans `NbPreferences` or any other persistent storage. When the report wizard is closed, these fields are garbage-collected with the object. When Autopsy is restarted, the input fields are empty and must be re-entered. The gateway URL and toggle states (upload/proposal enablement) are persisted — but these are configuration parameters, not authorisation credentials.

**Rationality:**

This design follows the principle of **minimal credential exposure**. Credentials in persistent storage face two risks. First, old credentials may be unintentionally reused — for example, an examiner uses a token in one session, opens Autopsy next time and finds the field already filled, and submits without generating a fresh token. Second, persistent storage (`NbPreferences` on Windows is typically stored in the registry or `%APPDATA%`) may be readable by other processes when Autopsy is not running. Confining credentials to memory and limiting them to the session lifetime effectively controls both risks.

The non-persistence of credentials also carries an implicit design intent: requiring the examiner to re-acquire and re-enter credentials for each upload. This institutionalised requirement of "active input each time" increases the operator's **awareness** of the upload action — the user must consciously recognise that "I am about to perform an upload operation" and take the explicit action of entering credentials, rather than clicking "Next" mindlessly through the report wizard.

### Decision 5: Local Report Generation Before Upload — Failure Isolation

**Design implementation:** In the workflow, the local report generation of Section 4.3.3 (`generateReport()` → write `case_data_extract.json` → register via `openCase.addReport()`) always executes before any upload/proposal HTTP request. Even if the gateway is unreachable, the token is invalid, or any network error causes the upload to fail, `case_data_extract.json` is already saved on disk and registered in the Autopsy case. The failure information is written to `upload_receipt.json` by `UploadReceiptWriter`, and `CaseEventRecorder.addEvent()` records `UPLOAD_FAILED` in the operation log, but the local report itself is not modified or removed.

**Rationality:**

This design establishes a clear hierarchical priority: the local forensic report is the **primary evidentiary artefact**, while the on-chain submission is a **secondary integrity commitment**. The failure of a secondary operation should not affect the integrity or accessibility of the primary artefact.

This priority reflects a fundamental principle of digital forensics: the existence and integrity of evidence should not depend on any external system. If the local report's existence were predicated on a successful upload (e.g., deleting the local report if the upload fails), then examiners working in an offline environment or during a network outage would lose all evidentiary records — which is unacceptable in forensic practice. Decoupling the two steps ensures the **independent survivability** of the local report — even if the gateway becomes permanently unavailable, the examiner still has a complete structured forensic report and local operation log.

Furthermore, `upload_receipt.json` records the failure details (error type, HTTP status code, error message, elapsed time), allowing the examiner to troubleshoot and resubmit without regenerating the report from scratch. This design embodies the principle of **not discarding completed partial work** in transactional persistence.

### Decision 6: Gateway-Side Pre-Chain Verification — Defence in Depth

**Design implementation:** Before invoking the smart contract to write to the chain, the gateway performs three sequential checks:

1. **Token verification**: Checks whether the one-time token in the `X-Auth-Token` HTTP header is valid and has not expired.
2. **Aggregate hash recomputation**: Performs canonical serialisation on `caseJson` → SHA-256 → 12-character truncation, and compares the result with the `aggregateHash` field in the request.
3. **Signing password verification** (contract mode): Pre-validates the signing password before invoking the `proposeUpdate` method of the FISCO BCOS contract.

Failure at any verification step directly returns an error response (with appropriate HTTP status code and error message) without executing any on-chain operation.

**Rationality:**

This design does not perform redundant verification on both the Autopsy side and the gateway side. Instead, it builds a **multi-layered defensive depth** (Defence in Depth). Relying solely on Autopsy-side verification is insufficient — if the Autopsy process has been compromised (e.g., malicious code modifies the output of `generateReport()` before calculating the hash), erroneous data could be written to the blockchain without external validation. The gateway, as an independent verification node running in a different process and (typically) a different machine from Autopsy, provides an independent judgement window.

Token verification prevents unauthorised third parties from exploiting a compromised gateway endpoint to perform write operations. Aggregate hash recomputation ensures the integrity of the submitted content — if the report file on Autopsy is silently modified between generation and upload (e.g., by a disk bad sector causing file corruption), the gateway's recomputed hash will not match, the request will be rejected, and corrupted data will not enter the on-chain record. Signing password verification adds a layer of human-authorised control over chain write operations — only the operator holding the signing password can trigger a blockchain state change.

The ordering of these three checks is also deliberate: the lightest check (token verification) executes first, and the heaviest executes last. If the token is invalid, the gateway can reject immediately without parsing the request body or computing hashes.

### Decision 7: Timing Instrumentation — Research-Oriented Observability

**Design implementation:** The configuration panel includes a "Collect timing metrics" checkbox. When enabled, the HTTP request includes an `X-Debug-Timing: 1` header. The gateway returns `integrityMs` (aggregate hash verification time), `chainMs` (CRUD table insertion time), `caseRegistryMs` (smart-contract execution time), and `totalMs` (total gateway-side time) in the response. On the Autopsy side, `UploadClientTiming` records `uploadStartedAt`, `uploadResponseAt`, and `clientRoundTripMs`. These timing fields are written into `upload_receipt.json`.

**Rationality:**

The timing instrumentation in the design does not serve a functional purpose for end users. Instead, it serves the evaluation needs of this research. Research questions RQ 2.1 (whether latency is acceptable), RQ 2.2 (latency distribution across stages), and RQ 2.3 (main bottlenecks) all require precise timing measurements. If the system design did not include built-in timing instrumentation, the evaluation phase would have to rely on external tools (e.g., tcpdump, Wireshark, manual timing) for coarse-grained measurements — not only lacking precision but also incapable of distinguishing the internal sub-stage latencies of the gateway.

In forensic practice, the most frequently questioned aspect of introducing blockchain into the workflow is the latency overhead — traditional hash verification is almost instantaneous, while blockchain transactions require waiting for consensus and block confirmation. The built-in timing data enables the researcher to answer the core question of "how many milliseconds each upload incurs and where the bottleneck lies", providing quantitative evidence for evaluating the system's feasibility in real forensic environments.

## 4. Summary of Overall Design Rationality

The Upload and Proposal Configuration module unifies three layers of considerations within a single mechanism. These three layers follow a natural progression: first, the architectural boundary that isolates the plugin from blockchain complexity; second, the workflow protections that ensure the primary forensic record survives operational failures; and third, the multi-party verification chain that makes the upload operation cryptographically provable.

**Operational boundary layer:** Through the gateway delegation pattern (Decision 1), the operational boundary between the Autopsy plugin and the blockchain infrastructure is clearly delineated. The plugin remains lightweight, chain interaction evolves independently, and cryptographic key management is centralised on the gateway. This boundary is also reflected in the deployment dimension — gateway updates do not require rebuilding or reinstalling the Autopsy plugin.

**Workflow integrity layer:** Through decisions such as mutually exclusive mode design (Decision 2), local report-first execution (Decision 5), and session-level credential handling (Decision 4), this layer ensures that the upload operation does not compromise the integrity of the primary evidentiary artefact, does not leave credentials in persistent storage that could be unintentionally reused, and does not confuse different chain-of-custody procedural stages. These decisions collectively form a mechanism that **protects forensic workflow integrity** — even if the upload fails, the local record remains intact; even if Autopsy is closed, credentials do not persist to the next session.

**Verifiability layer:** The self-contained verification payload (Decision 3) and the pre-chain verification checks (Decision 6) together construct a complete verification chain: from the **Autopsy local report**, to the **gateway verification result**, to the **on-chain hash commitment**. Every participant along this chain has independent verification capability — Autopsy can verify the integrity of the local report, the gateway can verify the integrity of the submitted content, and future auditors can verify through the on-chain hash whether the report has been tampered with. The timing instrumentation (Decision 7), while not part of the verification chain itself, provides the empirical observability needed to evaluate whether the latency introduced by these verification layers is acceptable in practice.

The superposition of these three layers ensures that the upload operation itself becomes trustworthy — not only is the report content delivered to the gateway, but the delivery process itself is verified, recorded, and cryptographically provably linked to the hash anchored on the blockchain. This is highly consistent with the requirement of ISO/IEC 27037 for the preservation phase after digital evidence acquisition — that integrity verification should span the entire lifecycle.

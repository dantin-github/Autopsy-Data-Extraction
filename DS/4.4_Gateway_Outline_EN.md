# 4.4 API Gateway Implementation — Outline

> This section describes the gateway as a software module organised around four functional modules: authentication and role routing, upload and integrity verification, two-party approval workflow, and query/audit. Cross-language hash consistency is covered in a separate section. Blockchain smart-contract internals and the judge dashboard are excluded from this section.

## Positioning in Chapter 4

Sections 4.3.1–4.3.5 cover the Autopsy-side implementation. The local structured report (`case_data_extract.json`) generated in Section 4.3.3 enters the gateway workflow through the upload configuration layer described in Section 4.3.5. Section 4.4 shifts focus to the server-side gateway that receives this report, authenticates the operator, verifies integrity, manages the modification-approval process, and provides query and audit services.

The gateway serves two client types: the Autopsy plugin (unattended HTTP requests with `X-Auth-Token`) and the judge web dashboard (browser-session-based). This section covers the gateway's own implementation across four functional modules.

---

## 4.4.1 Module 1: Authentication and Role Routing

### 4.4.1.1 Functional Role

The authentication module controls who can access each gateway function. It separates two operator roles — police and judge — with distinct authentication paths and permission boundaries. The police role is responsible for evidence submission and modification proposal initiation; the judge role is responsible for case querying, proposal approval/rejection, and audit log viewing.

### 4.4.1.2 Police Authentication: Two-Factor with OTP

The police authentication path uses two factors. The first factor is password verification via `bcrypt.compare()` against `data/users.json`. After the password is verified, the gateway generates a 16-byte random hex string as a one-time password (OTP) and delivers it through `nodemailer`. During development, a `MAIL_DRY_RUN` mode prints the OTP to the gateway log instead of sending it over SMTP.

The OTP is stored in an in-memory `Map` with a configurable time-to-live (default 2 hours). It is presented to the gateway through the `X-Auth-Token` HTTP header. Two consumption modes are supported:

- **Single-use mode** (default): the token is invalidated after the first successful use, preventing replay attacks.
- **Reusable mode**: the token can be used multiple times within its TTL, supporting batch-upload scenarios.

A police browser session can be obtained by exchanging the OTP through `POST /api/auth/police-otp`. This session is used for proposal initiation and execution through a web interface.

### 4.4.1.3 Judge Authentication: Password and Session

Judge authentication is simpler: password verification via `bcrypt` followed by session creation. No OTP is required. The judge session is an `express-session` cookie (`gw.sid`, with `httpOnly` and `sameSite: lax`). After successful login, the gateway redirects the judge to the dashboard URL (default `http://localhost:8501`). Session timeout defaults to 2 hours.

### 4.4.1.4 Authentication Middleware Chain

The gateway implements five authentication middlewares:


| Middleware                       | Auth mechanism                    | Sets on `req`                           | Used by endpoints                                                       |
| -------------------------------- | --------------------------------- | --------------------------------------- | ----------------------------------------------------------------------- |
| `requirePoliceToken`             | `X-Auth-Token` header             | `req.policeUserId`                      | `/api/upload`, `/api/modify/propose-with-token`                         |
| `requirePoliceSession`           | `req.session.role === 'police'`   | `req.policeUserId`                      | `/api/modify/propose`, `/api/modify/execute`                            |
| `requireJudgeSession`            | `req.session.role === 'judge'`    | `req.judgeUserId`                       | `/api/query`, `/api/modify/approve`, `/api/modify/reject`, `/api/audit` |
| `requireAnySession`              | Any logged-in session             | `req.policeUserId` or `req.judgeUserId` | `/api/modify/:proposalId`                                               |
| `requirePoliceTokenOrAnySession` | Token header, fallback to session | `req.policeUserId`                      | `/api/case-exists/:caseId`                                              |


### 4.4.1.5 Design Rationale

The separation between token-based and session-based authentication is driven by the two client types. The Autopsy plugin uses `java.net.HttpURLConnection`, which cannot maintain a browser session, so it relies on `X-Auth-Token`. The judge dashboard is a standard web application and uses cookie-based sessions.

OTP is chosen over long-lived API keys because it ties each submission to a fresh authorisation event. The single-use/reusable dual mode provides flexibility: single-use for normal operation, reusable for batch uploads.

The police/judge role separation reflects organisational separation of duties in forensic practice. No single operator can both submit evidence and approve its modification.

## 4.4.2 Module 2: Upload and Integrity Verification

### 4.4.2.1 Functional Role

This module receives the structured report from Autopsy, validates the submission, recomputes the integrity hash to verify that the report has not been altered during transmission, computes the cryptographic identifiers needed for downstream storage, and records the result.

### 4.4.2.2 Upload Request Contract

The upload endpoint (`POST /api/upload`) accepts a JSON body with five fields:


| Field           | Type                 | Purpose                                               |
| --------------- | -------------------- | ----------------------------------------------------- |
| `caseId`        | string               | Identifies the investigation case                     |
| `examiner`      | string               | Records the operator who performed the submission     |
| `aggregateHash` | string (12-char hex) | Report-level integrity commitment computed by Autopsy |
| `generatedAt`   | string (ISO 8601)    | Timestamp of report generation on Autopsy             |
| `caseJson`      | string               | Full text of `case_data_extract.json`                 |


The `caseJson` field contains the entire local report. This is a deliberate design: it allows the gateway to independently recompute the aggregate hash from the received content rather than trusting a hash value alone.

### 4.4.2.3 Upload Processing Pipeline

**Stage 1 — Input validation:** The gateway checks all required fields, validates the token, and checks whether the `caseId` already exists in the record store.

**Stage 2 — Aggregate hash recomputation:** The gateway calls `integrity.verify(caseJson, aggregateHash)`. This function parses `caseJson` as JSON, removes metadata fields (`aggregateHash`, `aggregateHashNote`, `uploadStatus`, `uploadDetail`), sorts remaining keys recursively, serialises without extra whitespace, computes SHA-256, and compares the first 12 hex characters with the submitted `aggregateHash`. If they do not match, the gateway returns `400` with error kind `aggregate_hash_mismatch`.

**Stage 3 — Hash computation:** The gateway computes `indexHash = SHA256(caseId)` and `recordHash = SHA256(caseId + examiner + aggregateHash + generatedAt + caseJson)`. These serve as the lookup key and cryptographic commitment respectively.

**Stage 4 — Downstream write:** The gateway delegates the hash records to the configured storage layer (CRUD table, smart contract, or both). The implementation of this delegation is covered in the blockchain section.

**Stage 5 — Local record and response:** The gateway updates the local record store (`~/.case_record_store.json`) with the `caseJson` content. The response includes `indexHash`, `recordHash`, transaction identifiers, block number, and optional timing fields.

### 4.4.2.4 Error Classification


| HTTP status | Error kind                 | Trigger                                                        |
| ----------- | -------------------------- | -------------------------------------------------------------- |
| 400         | `aggregate_hash_mismatch`  | Re-computed hash does not match submitted value                |
| 400         | `signing_password_missing` | Contract mode requires signing password but none provided      |
| 400         | `duplicate_case`           | Case ID already registered                                     |
| 401         | `token_invalid`            | OTP does not exist or has expired                              |
| 401         | `token_consumed`           | OTP already used in single-use mode                            |
| 403         | `forbidden`                | Authenticated user lacks required role                         |
| 503         | `chain_unavailable`        | Downstream blockchain service is not configured or unreachable |


### 4.4.2.5 Timing Instrumentation

The gateway includes a timing mechanism that records phase-level latency when the `X-Debug-Timing: 1` header is present:

- `integrityMs`: aggregate hash recomputation time
- `chainMs`: downstream storage write time
- `caseRegistryMs`: contract-level write time (when applicable)
- `totalMs`: total gateway processing time
- `requestId`: UUID for correlating logs with client receipts
- `blockTimestampUtc`: on-chain block timestamp

Timing is disabled by default and controlled by `UPLOAD_TIMING_IN_RESPONSE`. It is designed for research evaluation (RQ 2.1, RQ 2.2, RQ 2.3) rather than production monitoring.

## 4.4.3 Module 3: Two-Party Approval Workflow

### 4.4.3.1 Functional Role

After a case record has been submitted to the gateway, subsequent analysis may discover new evidence that needs to be appended or corrected. The modification proposal workflow allows a police operator to propose changes to an existing case record. These proposals must pass through a judge's approval before the changes are applied. This prevents a single operator from unilaterally modifying an evidentiary record after it has been submitted.

### 4.4.3.2 Workflow Endpoints


| Method | Path                                   | Auth           | Action                                         |
| ------ | -------------------------------------- | -------------- | ---------------------------------------------- |
| POST   | `/api/modify/propose`                  | police session | Create proposal with new `caseJson` and reason |
| POST   | `/api/modify/propose-with-token`       | police token   | Same, for Autopsy HTTP client                  |
| POST   | `/api/modify/approve`                  | judge session  | Approve pending proposal                       |
| POST   | `/api/modify/reject`                   | judge session  | Reject pending proposal                        |
| POST   | `/api/modify/execute`                  | police session | Apply approved proposal to record store        |
| GET    | `/api/modify/pending-for-case/:caseId` | judge session  | List pending proposals for a case              |
| GET    | `/api/modify/:proposalId`              | any session    | Query proposal detail                          |


### 4.4.3.3 Proposal Lifecycle

The proposal lifecycle consists of three phases:

**Phase 1 — Proposal creation (police):** The police operator submits a proposal containing a new `caseJson` (the updated report) and a human-readable reason explaining why the modification is necessary. The gateway assigns a `proposalId` and records the proposal in the local record store under a `pending-{proposalId}` key. The original case record is preserved unchanged.

**Phase 2 — Approval or rejection (judge):** A judge reviews the pending proposal. The judge can either approve or reject it. Rejection terminates the proposal; approval moves it to the next phase. The proposal reason provides the context needed for the judge's review.

**Phase 3 — Execution (police or automatic):** After approval, the police operator (or the automatic execution mechanism) applies the proposed changes. The local record store is updated: the original case record is replaced with the proposed `caseJson`, and the pending entry is removed.

### 4.4.3.4 Proposal Reason

The proposal reason is a mandatory free-text field. It serves as a human-readable audit record explaining why the modification was necessary — for example, "new hard drive discovered during secondary search" or "correction of misattributed file hash." This reason is preserved in the audit trail alongside the proposal's lifecycle events.

### 4.4.3.5 State Constraints

The gateway enforces several constraints at the application layer:

- Only registered police operators can initiate proposals.
- Only registered judges can approve or reject proposals.
- A judge cannot approve their own proposal (this is also enforced at the smart-contract layer).
- Only one pending proposal is allowed per case at a time.
- An approved proposal can only be executed once.
- A rejected proposal cannot be executed or re-approved.

### 4.4.3.6 Design Rationale

The two-party approval workflow addresses a gap in traditional forensic practice: once evidence data has been submitted to a custody system, there is often no procedural mechanism to handle the discovery of new or corrected information without overwriting the original record. The gateway's proposal workflow provides an audited path for modification: the original record is preserved as a baseline, the proposed change is recorded with a reason, and a second party (judge) must review and approve before the change is applied.

This is directly relevant to RQ 1.3, which asks whether runtime data — including modification attempts — can be systematically collected and managed. The proposal workflow collects the operator identity, proposal reason, timing, and outcome of each modification attempt as part of the gateway's audit record.

## 4.4.4 Module 4: Query and Audit

### 4.4.4.1 Functional Role

The query and audit module provides judges with the ability to retrieve case records and review the event history. It serves as the information retrieval interface for the judge web dashboard.

### 4.4.4.2 Case Existence Check

`GET /api/case-exists/:caseId` is a lightweight endpoint. It checks whether a case has been previously submitted to the gateway by looking up the derived `indexHash`. It returns a boolean result. This endpoint is used by the Autopsy plugin before submitting a modification proposal to confirm that the target case exists.

### 4.4.4.3 Full Case Query

`POST /api/query` accepts a `caseId` in the request body and returns the full case record along with verification results. The response includes:

- The `caseJson` content from the local record store
- Verification status: whether the stored aggregate hash matches a fresh recomputation from the stored `caseJson`
- Chain-side comparison: whether the stored `recordHash` matches the on-chain record
- Reconciliation status: whether data was successfully retrieved from all expected sources

Only judges can access this endpoint. It provides the primary evidence review interface for the judge dashboard.

### 4.4.4.4 Audit Log Query

`GET /api/audit` provides access to the audit event log. It supports two query parameters:

- `limit` (default 50, maximum 500): controls the number of returned events
- `since` (ISO 8601 timestamp or Unix milliseconds): filters events after a given time

The audit log is stored as JSONL (`data/audit.jsonl`), with each line representing one blockchain event. The response includes enriched entries where raw blockchain addresses are mapped to operator aliases for readability.

### 4.4.4.5 Error Response Structure

All endpoints return a consistent error structure: `{ error: "<message>", code: "<errorKind>" }`. For errors that originate from downstream blockchain calls, an additional `chainError` field may be present with `revertReason` and `txHash` to help operators diagnose the root cause.

## 4.4.5 Summary of Design Rationality

The gateway implementation is structured around four modules that correspond to four stages of the evidence management workflow:

**Authentication and role routing (Module 1)** gates access at the entry point. Before any evidence operation, the operator must be authenticated and the system must confirm that the operator has the correct role for the requested action. The two-factor design for police and the simpler session-based design for judges reflect the different operational contexts in which each role works.

**Upload and integrity verification (Module 2)** gates the data entering the system. The gateway does not trust the hash value submitted by Autopsy without independent recomputation. By re-parsing `caseJson`, re-applying the canonical serialisation rule, and re-computing SHA-256, the gateway ensures that the data it receives is byte-for-byte identical to what Autopsy generated. The timing instrumentation in this module provides the empirical observability needed to evaluate the latency introduced by the verification process.

**Two-party approval workflow (Module 3)** gates changes to existing records. After a case record has been submitted, any modification must follow a three-phase lifecycle: police proposal, judge review (approval or rejection), and execution. This prevents unilateral modification and creates an auditable history of why each change was made.

**Query and audit (Module 4)** provides the information retrieval interface. Judges can verify the integrity of a submitted case, review the full case content, and inspect the event history. The existence check endpoint also supports the Autopsy plugin's pre-proposal validation.

Together, these four modules define the gateway as an active verification and governance layer between Autopsy and the downstream blockchain infrastructure, rather than a passive relay.

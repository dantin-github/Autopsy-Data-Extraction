# 4.4 API Gateway 实现——论文大纲

> 本节按四个功能模块组织 Gateway：认证分流 → 上传完整性校验 → 两方审批工作流 → 查询/审计。跨语言哈希一致性由专门章节覆盖，区块链智能合约细节和法官面板不在此处展开。

## 在第四章中的定位

4.3.1–4.3.5 覆盖 Autopsy 端实现。第 4.3.3 节生成本地结构化报告（`case_data_extract.json`），第 4.3.5 节的上传配置层将其送入 Gateway 工作流。第 4.4 节将焦点转向服务端 Gateway，它接收该报告、认证操作者、校验完整性、管理修改审批流程，并提供查询与审计服务。

Gateway 服务两类客户端：Autopsy 插件（携带 `X-Auth-Token` 的无人值守 HTTP 请求）和法官 Web Dashboard（基于浏览器会话）。本节覆盖 Gateway 自身的实现，按四个功能模块组织。

---

## 4.4.1 模块一：认证分流（Authentication & Role Routing）

### 4.4.1.1 功能定位

认证模块控制谁能访问 Gateway 的哪些功能。它将操作者区分为两个角色——警察（police）和法官（judge），分别具有不同的认证路径和权限边界。警察角色负责证据提交和修改提案的发起；法官角色负责案件查询、提案审批/驳回以及审计日志查看。

### 4.4.1.2 警察认证：双因子 OTP

警察认证路径使用两个因子。第一因子是密码验证，通过 `bcrypt.compare()` 对 `data/users.json` 中的记录进行验证。密码以 bcrypt 哈希而非明文存储。第二因子是 OTP：密码验证后，Gateway 生成 16 字节随机 hex 字符串作为一次性密码，通过 `nodemailer` 投递到操作者邮箱。开发测试期间可通过 `MAIL_DRY_RUN` 模式将 OTP 打印到日志而非实际发送邮件。

OTP 存储在内存 `Map` 中，带有可配置生存时间（默认 2 小时）。通过 HTTP 头 `X-Auth-Token` 提交给 Gateway。支持两种消耗模式：

- **单次使用模式（默认）**：令牌在首次成功使用后失效，防止重放攻击。
- **可复用模式**：令牌在 TTL 内可多次使用，支持批量上传场景。

警察浏览器会话可通过 `POST /api/auth/police-otp` 将 OTP 兑换为 `express-session` cookie。该会话用于通过 Web 界面进行提案发起和执行。

### 4.4.1.3 法官认证：密码 + 会话

法官认证路径更简洁：通过 `bcrypt` 密码验证后直接创建基于 cookie 的会话。无需 OTP。法官会话是标准 `express-session` cookie（`gw.sid`，配置为 `httpOnly` 和 `sameSite: lax`）。登录成功后 Gateway 将法官重定向到 Dashboard URL（默认 `http://localhost:8501`）。会话超时默认为 2 小时。

### 4.4.1.4 认证中间件链

Gateway 实现了五个认证中间件，分别覆盖不同的认证上下文：


| 中间件                              | 认证方式                            | 在 `req` 上设置                            | 用于的端点                                                                |
| -------------------------------- | ------------------------------- | -------------------------------------- | -------------------------------------------------------------------- |
| `requirePoliceToken`             | `X-Auth-Token` 头                | `req.policeUserId`                     | `/api/upload`、`/api/modify/propose-with-token`                       |
| `requirePoliceSession`           | `req.session.role === 'police'` | `req.policeUserId`                     | `/api/modify/propose`、`/api/modify/execute`                          |
| `requireJudgeSession`            | `req.session.role === 'judge'`  | `req.judgeUserId`                      | `/api/query`、`/api/modify/approve`、`/api/modify/reject`、`/api/audit` |
| `requireAnySession`              | 任意已登录会话                         | `req.policeUserId` 或 `req.judgeUserId` | `/api/modify/:proposalId`                                            |
| `requirePoliceTokenOrAnySession` | 令牌头，回退到会话                       | `req.policeUserId`                     | `/api/case-exists/:caseId`                                           |


### 4.4.1.5 设计理由

令牌认证与会话认证的分离由两种客户端类型驱动。Autopsy 插件通过 `java.net.HttpURLConnection` 发送 HTTP 请求，无法维护浏览器会话，因此依赖每次请求携带的 `X-Auth-Token`。法官 Web Dashboard 是标准 Web 应用，使用基于 cookie 的会话。

选择 OTP 而非长生命周期的 API Key，是因为它将每次提交绑定到一次新的授权事件。单次使用/可复用的双模式提供灵活度：常规操作使用单次使用模式，批量上传使用可复用模式。

警察/法官的角色分离反映取证实践中的组织职责分离原则——单一操作者不能既提交证据又审批对证据记录的修改。

## 4.4.2 模块二：上传完整性校验（Upload Integrity Verification）

### 4.4.2.1 功能定位

本模块接收来自 Autopsy 的结构化报告、验证提交合法性、重新计算完整性哈希以确认报告在传输过程中未被篡改、计算下游存储所需的密码学标识符，并记录结果。

### 4.4.2.2 上传请求约定

上传端点（`POST /api/upload`）接受包含五个字段的 JSON 请求体：


| 字段              | 类型                   | 用途                             |
| --------------- | -------------------- | ------------------------------ |
| `caseId`        | string               | 标识调查案件                         |
| `examiner`      | string               | 记录执行提交的操作者                     |
| `aggregateHash` | string (12-char hex) | Autopsy 计算的报告级完整性承诺            |
| `generatedAt`   | string (ISO 8601)    | Autopsy 上报告生成的时间戳              |
| `caseJson`      | string               | `case_data_extract.json` 的完整文本 |


`caseJson` 字段包含完整的本地报告。这是有意设计：它使 Gateway 能够从接收到的内容独立重新计算聚合哈希，而不是仅信任提交的哈希值。

### 4.4.2.3 上传处理流水线

**第一阶段——输入验证：** Gateway 检查所有必填字段是否存在，验证令牌有效性，检查 `caseId` 是否已存在于记录存储中（防止重复提交）。

**第二阶段——聚合哈希重算：** Gateway 调用 `integrity.verify(caseJson, aggregateHash)`。该函数解析 `caseJson` 为 JSON，移除元数据字段（`aggregateHash`、`aggregateHashNote`、`uploadStatus`、`uploadDetail`），递归排序剩余键，无多余空格序列化，计算 SHA-256，取前 12 个 hex 字符与提交的 `aggregateHash` 比较。若不匹配则返回 `400`，error kind 为 `aggregate_hash_mismatch`，确保收到的内容与 Autopsy 生成的内容逐字节一致。

**第三阶段——哈希计算：** Gateway 计算两个哈希：`indexHash = SHA256(caseId)` 和 `recordHash = SHA256(caseId + examiner + aggregateHash + generatedAt + caseJson)`。`indexHash` 是后续查找案件的键，`recordHash` 是将提交参数绑定在一起的密码学承诺。

**第四阶段——下游写入：** Gateway 将哈希记录委托给配置的存储层（CRUD 表、智能合约或两者）。该委托的细节在区块链实现章节讨论。

**第五阶段——本地记录与响应：** Gateway 用 `caseJson` 内容更新本地记录存储（`~/.case_record_store.json`）。响应包含 `indexHash`、`recordHash`、交易标识符、区块号和可选的计时字段。

### 4.4.2.4 错误分类


| HTTP 状态 | Error kind                 | 触发条件             |
| ------- | -------------------------- | ---------------- |
| 400     | `aggregate_hash_mismatch`  | 重新计算的哈希与提交值不匹配   |
| 400     | `signing_password_missing` | 合约模式需要签名密码但未提供   |
| 400     | `duplicate_case`           | 案件 ID 已注册        |
| 401     | `token_invalid`            | OTP 不存在或已过期      |
| 401     | `token_consumed`           | OTP 在单次使用模式下已被使用 |
| 403     | `forbidden`                | 已认证用户缺少所需角色      |
| 503     | `chain_unavailable`        | 下游区块链服务未配置或不可达   |


### 4.4.2.5 计时仪器

Gateway 内置计时机制，在 `X-Debug-Timing: 1` 头存在时记录阶段级延迟：

- `integrityMs`：聚合哈希重算时间
- `chainMs`：下游存储写入时间
- `caseRegistryMs`：合约级写入时间（如适用）
- `totalMs`：Gateway 端总处理时间
- `requestId`：用于关联日志与客户端回执的 UUID
- `blockTimestampUtc`：链上区块时间戳

计时默认禁用，由 `UPLOAD_TIMING_IN_RESPONSE` 控制。专为研究评估（RQ 2.1、RQ 2.2、RQ 2.3）设计。

## 4.4.3 模块三：两方审批工作流（Two-Party Approval Workflow）

### 4.4.3.1 功能定位

案件记录提交到 Gateway 后，后续分析可能发现需要追加或更正的新证据。修改提案工作流允许警察操作者对已有案件记录提议修改。这些提案必须经过法官审批才能生效，防止单一操作者在记录提交后单方面修改证据。

### 4.4.3.2 工作流端点


| 方法   | 路径                                     | 认证             | 动作                      |
| ---- | -------------------------------------- | -------------- | ----------------------- |
| POST | `/api/modify/propose`                  | police session | 用新的 `caseJson` 和理由创建提案  |
| POST | `/api/modify/propose-with-token`       | police token   | 同上，供 Autopsy HTTP 客户端使用 |
| POST | `/api/modify/approve`                  | judge session  | 审批待处理的提案                |
| POST | `/api/modify/reject`                   | judge session  | 驳回待处理的提案                |
| POST | `/api/modify/execute`                  | police session | 将已批准的提案应用到记录存储          |
| GET  | `/api/modify/pending-for-case/:caseId` | judge session  | 列出案件下待处理的提案             |
| GET  | `/api/modify/:proposalId`              | any session    | 查询提案详情                  |


### 4.4.3.3 提案生命周期

提案生命周期包含三个阶段：

**阶段 1——提案创建（警察）：** 警察操作者提交提案，包含新的 `caseJson`（更新后的报告）和人工可读的修改理由。Gateway 分配 `proposalId`，以 `pending-{proposalId}` 键存入本地记录存储。原案件记录保持不变。

**阶段 2——审批或驳回（法官）：** 法官审查待处理的提案。驳回则终止提案；批准则进入下一阶段。提案理由为法官审查提供上下文。

**阶段 3——执行（警察或自动）：** 提案批准后，警察操作者（或自动执行机制）应用修改。本地记录存储更新：原案件记录替换为提案中的 `caseJson`，待处理条目移除。

### 4.4.3.4 提案理由

提案理由是必填的文本字段。它作为审计记录中人工可读的字段，解释为何需要修改——例如"二次搜索中发现新硬盘"或"更正错误归因的文件哈希"。该理由与提案生命周期事件一起保留在审计追踪中。

### 4.4.3.5 状态约束

Gateway 在应用层施加若干约束：

- 只有已注册的警察才能发起提案。
- 只有已注册的法官才能审批或驳回提案。
- 法官不能审批自己的提案（智能合约层也施加此约束）。
- 每个案件同一时间只能有一个待处理提案。
- 已批准的提案只能执行一次。
- 已驳回的提案不能执行或重新审批。

### 4.4.3.6 设计理由

两方审批工作流解决传统取证实践中的一个缺口：证据数据提交到监管系统后，当发现新信息或需更正时，往往缺乏覆写原记录的程序化机制。Gateway 的提案工作流提供了一条可审计的修改路径：原始记录作为基线保留，提议的修改记录理由，必须由第二方（法官）审查和批准后才能执行。

这与 RQ 1.3 直接相关——该问题询问运行时数据（包括修改尝试）能否被系统性地收集和管理。提案工作流将操作者身份、提案理由、时间线和每次修改尝试的结果作为 Gateway 审计记录的一部分收集。

## 4.4.4 模块四：查询/审计（Query & Audit）

### 4.4.4.1 功能定位

查询与审计模块为法官提供检索案件记录和审查事件历史的能力。它是法官 Web Dashboard 的信息检索接口。

### 4.4.4.2 案件存在性检查

`GET /api/case-exists/:caseId` 是一个轻量级端点。它通过查询 `indexHash` 检查案件是否已提交到 Gateway，返回布尔值。Autopsy 插件在提交修改提案前使用此端点确认目标案件存在。

### 4.4.4.3 完整案件查询

`POST /api/query` 接受请求体中的 `caseId`，返回完整案件记录及校验结果。响应内容包括：

- 本地记录存储中的 `caseJson` 内容
- 校验状态：存储的聚合哈希与从存储的 `caseJson` 重新计算的哈希是否匹配
- 链端比对：存储的 `recordHash` 与链上记录是否一致
- 对账状态：是否从所有预期来源成功检索到数据

仅法官可访问此端点。它为法官 Dashboard 提供主要的证据审查接口。

### 4.4.4.4 审计日志查询

`GET /api/audit` 提供审计事件日志的访问。支持两个查询参数：

- `limit`（默认 50，最大 500）：控制返回事件数
- `since`（ISO 8601 时间戳或 Unix 毫秒）：过滤给定时间之后的事件

审计日志存储为 JSONL（`data/audit.jsonl`），每行代表一个区块链事件。响应包含富化后的事件条目，原始区块链地址映射为操作者别名以提高可读性。

### 4.4.4.5 错误响应结构

所有端点使用一致的错误结构：`{ error: "<message>", code: "<errorKind>" }`。对于源自下游区块链调用的错误，可能额外包含 `chainError` 字段，内含 `revertReason` 和 `txHash`，帮助操作者诊断根本原因。

## 4.4.5 整体设计合理性总结

Gateway 的实现围绕四个模块组织，分别对应证据管理流程的四个阶段：

**认证分流（模块一）** 在入口处控制访问。任何证据操作之前，操作者必须被认证，系统必须确认操作者拥有所请求操作的正确角色。警察的两因子设计和法官的简洁会话设计反映各自不同的操作上下文。

**上传完整性校验（模块二）** 在数据进入系统时进行验证。Gateway 不会盲目信任 Autopsy 提交的哈希值，而是从接收到的内容独立重新计算聚合哈希。计时仪器提供评估验证过程延迟所需的过程级可观测性。

**两方审批工作流（模块三）** 控制对已有记录的修改。案件记录提交后，任何修改必须遵循三阶段生命周期：警察提案、法官审查（审批或驳回）、执行。这防止单方面修改，并为每次变更建立可审计的原因记录。

**查询/审计（模块四）** 提供信息检索接口。法官可以校验已提交案件的完整性、审查完整案件内容、检查事件历史。存在性检查端点也为 Autopsy 插件的提案前验证提供支持。

这四个模块共同将 Gateway 定义为 Autopsy 与下游区块链基础设施之间的主动验证与治理层，而非被动中继。
